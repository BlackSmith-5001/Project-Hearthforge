# Response Change Bands 設計メモ

この文書は、旧CLI表現を Response Change Bands（RCB / 応答変化帯域）へ移行するための、v1.5.2候補の設計メモです。

これはまだ正式ガイドではありません。この段階では、JSONテンプレート、README、Glossary、既存の公開用語を変更せず、v1.6.0に向けた安全な語彙を準備します。

## 1. CLIからRCBへ移行する理由

旧称 CLI（Cognitive Load Index）には、いくつかの問題があります。

- CLI は Command Line Interface と誤読されやすい。
- Cognitive Load Index は、AIモデル内部の認知負荷を測るように見える。
- Index は精密指標を連想させる。
- 0-100% 表現は、実際以上の精密さを感じさせる。

旧称CLIは履歴として残しつつ、今後の公開ガイドでは、より安全な運用語へ移行するのが望ましいです。

## 2. RCBの定義

RCB は Response Change Bands（応答変化帯域）を意味します。

RCB は、アシスタントの出力に観測される変化を見るためのものです。モデル内部状態、感情、疲労、意識、記憶を診断するものではありません。

RCB は Response Change Catalog（応答変化カタログ）と対になります。

- カタログは、観測できる応答変化の種類を整理する。
- RCB は、その短期的な強さを粗い帯域として扱う。

## 3. 観測カウント

RCB はパーセンテージではなく、小さな観測カウントを使います。

カウント案:

- 応答変化の型が出た: +1
- すでに示した前提の保持に失敗した: +1
- 会話リズムが明確に破綻した: +2
- 最大値: 4

このカウントは、短期的な会話上の合図として扱います。

- 1応答だけで判断しない。
- 2〜3応答の傾向を見る。
- 会話ごとにリセットする。
- AIの通算健康スコアとして扱わない。

## 4. 0〜4帯域案

| カウント | 帯域 | 意味 |
| --- | --- | --- |
| 0 | 澄み | 目立つ応答変化は見られない。 |
| 1 | さざ波 | 小さな応答変化はあるが、会話はおおむね安定している。 |
| 2 | 揺らぎ | 複数の小さな変化、または意味のある不一致が見える。 |
| 3 | 乱れ | 応答パターンがはっきり不安定、または期待から外れている。 |
| 4 | 嵐 | そのまま押し進めず、会話を止めるか再読込を検討する。 |

これらの名前は、ユーザー側の観測のための比喩です。AI内部状態の測定ではありません。

## 5. 各帯域でのユーザー側対処

| 帯域 | ユーザー側の対処 |
| --- | --- |
| 0 澄み | そのまま続ける。 |
| 1 さざ波 | 話題を広げすぎない。次の入力をシンプルにする。 |
| 2 揺らぎ | 話題を1つに絞り、短文化し、必要な前提を言い直す。 |
| 3 乱れ | 会話を区切り、Temporal Anchor（時刻の錨）を再提示するか、休憩する。 |
| 4 嵐 | セッションを終了するか翌日再開する。必要ならPIM-DBSプロファイルから再読込する。 |

## 6. 避ける表現

診断や内部測定に見える表現は避けます。

- AIが疲れている。
- AIの負荷。
- 思考の重さ。
- 測定。
- 診断。
- 負荷率0〜100%。
- AIに今の負荷を自己申告させる。

代わりに、出力を主語にした表現を使います。

- 応答に〜が見られる。
- 出力パターンが変化している。
- 会話リズムが崩れているように見える。
- 先に示した前提が保持されていない。
