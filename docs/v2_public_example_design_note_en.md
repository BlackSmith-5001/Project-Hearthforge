# PIM-DBS v2.0 Public Example Design Note

This document is a non-normative design note. It explores how public examples could be designed for a possible PIM-DBS v2.0 structure.

It is not a committed v2.0 specification. It does not introduce a v2.0 draft template, JSON Schema, `probes.json`, `packet.md`, or any implementation change.

## 1. Purpose

The purpose of this note is to define what a safe public example should look like if PIM-DBS v2.0 separates profile material into:

- `core`
- `journal`
- `charter`
- `calibration`

Public examples should help new users understand the structure without encouraging them to publish private instances.

## 2. Why Public Examples Need Special Care

PIM-DBS profiles can contain personal context, relationship history, response-style examples, and verification notes.

If public examples look too realistic, users may misunderstand what is safe to share. A public example should demonstrate structure, not expose real private context.

Public examples should not imply that PIM-DBS stores AI consciousness, a soul, emotions, persistent hidden memory, or real internal states.

## 3. Safe Public Example Principles

A safe public example should:

- use a fully fictional persona
- use fictional users, fictional episodes, and fictional verification probes
- avoid real names, real organizations, real locations, and real conversation logs
- mark any sample memories or episodes as fictional
- show `journal` as a structure without real private content
- use Dialogue Exemplars only when they are fully fictional
- keep voice and tone examples generic or fictional
- treat `calibration` as user-side verification and observation
- make clear that safety guidelines and platform terms are not overridden

When in doubt, keep the example simpler.

## 4. What May Be Included

Public examples may include:

- blank fields and placeholders
- fictional persona names
- fictional role definitions
- generic tone and style descriptions
- fictional Shared Episodes
- fictional Dialogue Exemplars
- generic Provenance labels
- fictional Restoration Verification probes
- generic Response Change Bands examples
- generic Model Notes
- public/private safety reminders

The safest examples are clearly fictional and lightweight.

## 5. What Must Not Be Included

Public examples must not include:

- real Dialogue Exemplars
- real conversation logs
- real voiceprints or voice-derived style data
- private memories
- real family or relationship records
- real names, usernames, workplaces, schools, addresses, or locations
- medical, legal, financial, workplace, school, or family circumstances
- private inside jokes or identifiable relationship context
- unverified AI self-claims presented as fact
- private instance data by default

These items may belong in a private instance, but they should not be used as public example material.

## 6. Fictional Persona Examples

Fictional personas are the recommended basis for public examples.

A fictional persona should:

- have a clearly fictional name
- avoid resembling a real person, real AI companion, or private relationship
- use a lightweight role, such as study helper, writing partner, or fictional guide
- avoid intimate, medical, workplace, family, or real-life dependency framing
- state that the example is not based on a real user or real conversation

Fictional examples can still show the structure of PIM-DBS without exposing private data.

## 7. Core in Public Examples

`core` is the safest v2.0 area for public examples.

Good public `core` content:

- fictional name
- fictional role
- purpose
- generic tone
- generic speaking style
- non-private boundaries

Avoid placing real relationship history, real voice-derived details, or private continuity cues in public `core` examples.

## 8. Journal in Public Examples

`journal` should be handled carefully in public examples.

For public examples, `journal` should usually show:

- structure only
- empty placeholders
- clearly fictional episodes
- notes that real journal contents are private-only

Public `journal` examples should not include real memories, real Dialogue Exemplars, private conversation excerpts, inside jokes, real family roles, or relationship history.

## 9. Charter in Public Examples

`charter` can be shown publicly when it is generic.

Good public `charter` content:

- operating principles
- communication boundaries
- publication reminders
- consent reminders
- safety notes

The charter is not a contract, command layer, or override of safety guidelines or platform terms. It is a user-side statement of intended operation and boundaries.

## 10. Calibration in Public Examples

`calibration` can be shown publicly when it uses generic or fictional checks.

Good public `calibration` content:

- fictional Restoration Verification probes
- expected response features based on visible output
- generic failure signals
- fictional Model Notes
- Temporal Anchor usage notes
- Response Change Bands examples based on observable response changes

Calibration should not diagnose AI internals. RCB is not a measurement of AI cognition, fatigue, emotion, consciousness, hidden memory, or model health.

## 11. Unsafe Example Patterns

Avoid public examples that:

- look like a real private profile with names changed
- include raw conversation excerpts from real chats
- preserve a real person's tone, voiceprint, or relationship pattern
- include family, workplace, school, medical, address, or real-name context
- imply that the AI remembers events independently
- ask the AI to self-report internal load, emotional state, or memory
- frame a private instance as something users should publish

If an example could make a real person recognizable, vulnerable, embarrassed, or exposed, it should not be public.

## 12. Recommended Minimal Public Sample

A minimal public v2.0 sample should use a fully fictional persona and include only:

- `core`: fictional identity, role, purpose, tone, and generic boundaries
- `journal`: empty structure or one clearly fictional episode
- `charter`: generic operating principles and safety reminders
- `calibration`: one or two fictional verification probes and generic RCB guidance

Recommended sample style:

- short
- obviously fictional
- beginner-readable
- free of private history
- explicit about public/private separation

The sample should demonstrate how the structure works, not how to publish a complete private instance.

## 13. Open Questions

Before creating v2.0 public examples, decide:

- Should v2.0 examples include a filled fictional `journal`, or only placeholders?
- How many fictional Dialogue Exemplars are safe and useful?
- Should public examples include family or relations fields only as empty placeholders?
- Should Model Notes be included in public examples, or kept in advanced examples?
- Should RCB examples be included directly in `calibration`, or referenced from guidance?
- How should examples warn users not to publish private instances?
